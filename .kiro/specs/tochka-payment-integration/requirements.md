# Требования: Интеграция с платёжным API ТОЧКА.БАНК

## Введение

Добавление на сайт ресторана Molo (molobistro.ru) интеграции с платёжным API ТОЧКА.БАНК для обработки онлайн-платежей через СБП (Система Быстрых Платежей). Реализация включает полный цикл работы с платежами: создание платёжных ссылок, получение информации о статусе платежей, обработку двухэтапных платежей (capture), возврат платежей, создание платежей с отправкой чека и получение реестра платежей.

## Глоссарий

- **ТОЧКА.БАНК** — банк-эквайер, предоставляющий API для приёма платежей через СБП.
- **Payment Operation** — платёжная операция в системе ТОЧКА.БАНК, идентифицируемая `payment_operation_id`.
- **Payment URL** — ссылка, которую клиент использует для оплаты через СБП.
- **Two-Stage Payment** — двухэтапный платёж: сначала авторизация (блокировка средств), затем списание (capture).
- **Capture** — списание ранее авторизованных средств при двухэтапном платеже.
- **Refund** — возврат ранее совершённого платежа.
- **Payment Registry** — реестр платежей, содержащий сводную информацию о платежах за период.
- **Webhook** — HTTP-запрос от ТОЧКА.БАНК с уведомлением об изменении статуса платежа.
- **PaymentService** — серверный модуль для работы с API ТОЧКА.БАНК.
- **OrderAPI** — серверный модуль Express для создания заказов и интеграции с PaymentService.
- **FiscalService** — серверный модуль для отправки чеков в облачную онлайн-кассу (54-ФЗ).
- **Redirect URL** — URL для перенаправления клиента после успешной или неуспешной оплаты.

## Требования

### Требование 1: Создание платёжной операции (Create Payment Operation)

**User Story:** Как посетитель сайта, я хочу оплатить заказ через СБП, чтобы подтвердить свой заказ и произвести оплату.

#### Критерии приёмки

1. WHEN пользователь подтверждает заказ, THE OrderAPI SHALL вызывать метод Create Payment Operation API ТОЧКА.БАНК с параметрами: сумма, номер заказа, описание, redirect URL.
2. THE PaymentService SHALL формировать запрос к API ТОЧКА.БАНК в соответствии со спецификацией API (https://developers.tochka.com/api/payments#create-payment-operation).
3. WHEN API возвращает успешный ответ, THE PaymentService SHALL извлекать `payment_operation_id` и `payment_url` из ответа.
4. THE PaymentService SHALL сохранять `payment_operation_id` в заказе в БД для последующего отслеживания.
5. WHEN клиент получает `payment_url`, THE CartUI SHALL перенаправлять пользователя на страницу оплаты ТОЧКА.БАНК.
6. IF API ТОЧКА.БАНК возвращает ошибку, THEN THE PaymentService SHALL логировать ошибку с полным описанием и возвращать понятное сообщение клиенту.
7. THE PaymentService SHALL обрабатывать таймауты при обращении к API ТОЧКА.БАНК и выполнять повторную попытку (retry) до 3 раз с экспоненциальной задержкой.
8. THE PaymentService SHALL использовать конфигурируемый endpoint API ТОЧКА.БАНК (тестовый или боевой) на основе переменной окружения `TOCHKA_API_ENV`.

### Требование 2: Получение списка платёжных операций (Get Payment Operation List)

**User Story:** Как администратор ресторана, я хочу видеть список всех платежей, чтобы отслеживать поступление оплат.

#### Критерии приёмки

1. THE AdminPanel SHALL предоставлять возможность получения списка платёжных операций через вызов API ТОЧКА.БАНК Get Payment Operation List.
2. THE PaymentService SHALL передавать параметры фильтрации: дата начала, дата окончания, статус платежа.
3. WHEN API возвращает список операций, THE PaymentService SHALL возвращать структурированный массив с полями: `payment_operation_id`, `status`, `amount`, `created_at`, `paid_at`.
4. THE AdminPanel SHALL отображать полученный список в таблице с возможностью сортировки по дате и статусу.
5. THE AdminPanel SHALL поддерживать пагинацию при большом количестве операций (не более 100 на странице).
6. IF API возвращает ошибку, THEN THE PaymentService SHALL кешировать последний успешный результат на 5 минут для повторного отображения без повторного запроса.

---

### Требование 3: Получение информации о платёжной операции (Get Payment Operation Info)

**User Story:** Как администратор ресторана, я хочу получить детальную информацию о конкретном платеже, чтобы проверить его статус и параметры.

#### Критерии приёмки

1. THE AdminPanel SHALL предоставлять возможность просмотра детальной информации о платеже по его `payment_operation_id`.
2. WHEN администратор запрашивает информацию о платеже, THE PaymentService SHALL вызывать метод Get Payment Operation Info API ТОЧКА.БАНК.
3. THE PaymentService SHALL возвращать и отображать все доступные поля: статус, сумма, валюта, дата создания, дата оплаты, способ оплаты, реквизиты плательщика, данные чека.
4. THE AdminPanel SHALL отображать историю изменения статусов платежа (если доступно от API).
5. IF платеж не найден (ошибка 404 от API), THEN THE PaymentService SHALL возвращать понятное сообщение "Платёж не найден" и HTTP 404.

### Требование 4: Двухэтапное списание средств (Capture Payment)

**User Story:** Как администратор ресторана, я х��чу выполнить списание ранее авторизованных средств, чтобы подтвердить оплату заказа.

#### Критерии приёмки

1. THE AdminPanel SHALL предоставлять функцию Capture для платежей со статусом `authorized` (авторизован, но не списан).
2. WHEN администратор нажимает "Списать средства", THE PaymentService SHALL вызывать метод Capture Payment API ТОЧКА.БАНК.
3. THE PaymentService SHALL передавать `payment_operation_id` и сумму списания (равную или меньше авторизованной).
4. WHEN списание успешно, THE PaymentService SHALL обновлять статус заказа в БД на `paid` и вызывать FiscalService для формирования чека.
5. IF списание неуспешно (ошибка от API), THEN THE PaymentService SHALL отображать сообщение об ошибке и сохранять её в логах.
6. THE AdminPanel SHALL отображать текущий статус платежа (authorized/captured/failed) и дату последнего изменения.
7. IF списание выполняется для частичной суммы, THEN THE PaymentService SHALL корректно обрабатывать partial capture согласно документации API.

---

### Требование 5: Возврат платежа (Refund Payment Operation)

**User Story:** Как администратор ресторана, я хочу оформить возврат платежа клиенту, чтобы обработать отмену заказа или возврат средств.

#### Критерии приёмки

1. THE AdminPanel SHALL предоставлять функцию возврата для платежей со статусом `paid` или `captured`.
2. WHEN администратор запрашивает возврат, THE AdminPanel SHALL отображать форму с выбором: полный возврат или частичный (с указанием суммы).
3. THE PaymentService SHALL вызывать метод Refund Payment Operation API ТОЧКА.БАНК с `payment_operation_id` и суммой возврата.
4. IF сумма возврата указана равной сумме платежа, THE PaymentService SHALL запрашивать полный возврат.
5. IF сумма возврата меньше суммы платежа, THE PaymentService SHALL запрашивать частичный возврат.
6. WHEN возврат успешно обработан, THE PaymentService SHALL обновлять статус заказа в БД на `refunded` или `partial_refunded`.
7. IF возврат неуспешен, THEN THE PaymentService SHALL отображать сообщение об ошибке от API.
8. THE AdminPanel SHALL логировать все операции возврата с указанием: администратора, даты, суммы, причины (если указана).
9. IF возврат выполняется более чем через 90 дней после платежа, THEN THE PaymentService SHALL предупреждать о возможных ограничениях со стороны банка.

### Требование 6: Создание платежа с чеком (Create Payment Operation WithReceipt)

**User Story:** Как администратор ресторана, я хочу автоматически отправлять чек при оплате, чтобы соблюдать требования 54-ФЗ.

#### Критерии приёмки

1. THE PaymentService SHALL поддерживать создание платежа с автоматической отправкой чека через метод Create Payment Operation WithReceipt.
2. THE PaymentService SHALL передавать в запросе данные чека: состав заказа (наименования, количество, цены), сумма НДС, данные продавца (ИНН, название).
3. THE PaymentService SHALL формировать данные чека в соответствии с требованиями ФНС (формат JSON для онлайн-кассы).
4. WHEN платёж успешно оплачен, THE FiscalService SHALL автоматически отправлять чек в облачную онлайн-кассу (если не передан через WithReceipt).
5. IF передача чека через WithReceipt недоступна, THEN THE PaymentService SHALL использовать отдельный вызов FiscalService после подтверждения оплаты.
6. THE PaymentService SHALL сохранять ссылку на чек (`fiscal_receipt_url`) в заказе при получении от API.

### Требование 7: Получение реестра платежей (Get Payment Registry)

**User Story:** Как администратор ресторана, я хочу получать реестр платежей за период, чтобы формировать отчётность и сверять платежи.

#### Критерии приёмки

1. THE AdminPanel SHALL предоставлять функцию получения реестра платежей с выбором периода (дата начала и дата окончания).
2. WHEN администратор запрашивает реестр, THE PaymentService SHALL вызывать метод Get Payment Registry API ТОЧКА.БАНК.
3. THE PaymentService SHALL передавать параметры: начальная дата, конечная дата, статус платежа (опционально).
4. THE PaymentService SHALL возвращать и AdminPanel SHALL отображать реестр в табличном формате с колонками: ID платежа, дата, сумма, статус, номер заказа.
5. THE AdminPanel SHALL рассчитывать и отображать итоговые суммы по реестру: общая сумма, сумма возвратов, чистая сумма поступлений.
6. THE AdminPanel SHALL предоставлять экспорт реестра в CSV/Excel формат для бухгалтерии.
7. IF период превышает 90 дней, THE PaymentService SHALL разбивать запрос на несколько с суточными интервалами.

---

### Требование 8: Обработка webhook-уведомлений от ТОЧКА.БАНК

**User Story:** Как система, я хочу получать уведомления об изменении статуса платежей в реальном времени, чтобы автоматически обновлять статусы заказов.

#### Критерии приёмки

1. THE OrderAPI SHALL предоставлять endpoint `/api/payment/webhook` для приёма уведомлений от ТОЧКА.БАНК.
2. WHEN webhook поступает, THE PaymentService SHALL верифицировать подпись запроса (если предоставлена ТОЧКА.БАНК).
3. THE PaymentService SHALL извлекать из webhook данные: `payment_operation_id`, новый статус, дату изменения.
4. WHEN получен webhook об успешной оплате (status = `paid`), THE PaymentService SHALL обновлять статус заказа в БД на `paid`.
5. WHEN получен webhook о неуспешной оплат�� (status = `failed`), THE PaymentService SHALL обновлять статус заказа на `failed`.
6. THE PaymentService SHALL логировать все входящие webhook-запросы с timestamp и телом запроса.
7. IF webhook содержит невалидные данные или неподтверждённая подпись, THEN THE PaymentService SHALL игнорировать запрос и логировать предупреждение.
8. THE PaymentService SHALL обрабатывать повторные webhook (idempotency) — при получении дубликата не изменять статус заказа повторно.

### Требование 9: Конфигурация и безопасность

**User Story:** Как администратор, я хочу безопасно настроить подключение к API ТОЧКА.БАНК, чтобы обеспечить защиту платёжных данных.

#### Критерии приёмки

1. THE PaymentService SHALL использовать OAuth-аутентификацию с `client_id` и `client_secret` для доступа к API ТОЧКА.БАНК.
2. THE PaymentService SHALL получать и обновлять access token автоматически (refresh token) перед каждым запросом.
3. THE PaymentService SHALL хранить `client_id`, `client_secret`, `refresh_token` в переменных окружения, недоступных извне.
4. THE PaymentService SHALL поддерживать переключение между тестовым (`sandbox`) и боевым (`production`) окружением через переменную `TOCHKA_API_ENV`.
5. THE OrderAPI SHALL логировать все запросы к API ТОЧКА.БАНК (без чувствительных данных) для отладки и аудита.
6. THE PaymentService SHALL использовать HTTPS для всех запросов к API ТОЧКА.БАНК.
7. THE AdminPanel SHALL предоставлять интерфейс для ввода и сохранения настроек подключения (client_id, client_secret) с маскированием пароля.

### Требование 10: Обработка ошибок и мониторинг

**User Story:** Как разработчик, я хочу видеть подробные ошибки при работе с API, чтобы быстро диагностировать и исправлять проблемы.

#### Критерии приёмки

1. THE PaymentService SHALL классифицировать ошибки API: сетевые ошибки, ошибки авторизации, ошибки валидации, ошибки бизнес-логики.
2. THE PaymentService SHALL возвращать разные HTTP-коды: 400 (ошибка валидации), 401 (ошибка авторизации), 404 (платёж не найден), 500 (внутренняя ошибка), 503 (временная недоступность API).
3. THE PaymentService SHALL логировать полный контекст ошибки: endpoint, параметры (без secrets), код ошибки, сообщение, stack trace.
4. THE AdminPanel SHALL отображать историю ошибок API с возможностью фильтрации по дате и типу ошибки.
5. THE PaymentService SHALL реализовывать circuit breaker: при 5 последовательных ошибках переключаться в режим ожидания на 30 секунд.
6. THE PaymentService SHALL отправлять alert администратору при критических ошибках (сбой авторизации, недоступность API более 1 минуты).

---

### Требование 11: Совместимость с существующим заказом

**User Story:** Как разработчик, я хочу, чтобы новая интеграция с ТОЧКА.БАНК работала с существующей моделью заказов без изменений в схеме БД.

#### Критерии приёмки

1. THE PaymentService SHALL расширять существующую таблицу `orders` колонками `payment_operation_id`, `payment_status`, `captured_at`, `refunded_at`, `refund_amount` если они ещё не существуют.
2. THE PaymentService SHALL сохранять обратную совместимость: все существующие вызовы OrderAPI (POST /api/orders, GET /api/orders) должны работать без изменений.
3. THE PaymentService SHALL использовать существующий webhook endpoint `/api/payment/webhook` для обратной совместимости.
4. THE CartUI SHALL продолжать работать с существующей логикой перенаправления на `payment_url`.
5. THE существующая админ-панель SHALL отображать новые поля платежа (payment_operation_id, payment_status) без изменений интерфейса.
