# Требования: Фискализация (54-ФЗ)

## Overview

Подключение облачной онлайн-кассы для формирования и отправки фискальных чеков в соответствии с требованиями 54-ФЗ. Чеки формируются автоматически при успешной оплате заказа.

## Scope

- Интеграция с облачной онлайн-кассой
- Автоматическая отправка чеков при оплате
- Отправка чеков возврата
- Отправка чеков клиенту на email
- Просмотр статуса чеков в админ-панели
- Повторная отправка чеков при ошибках

---

## Требования

### 19.1. Интеграция с облачной онлайн-кассой

**Описание**: Подключение к провайдеру фискальных данных (ПКФ).

**Детали**:
- Поддерживаемые провайдеры: CloudKassir, МойСклад, Эвотор, RetailCRM
- Настройка через переменные окружения:
  - `FISCAL_INN` — ИНН организации
  - `FISCAL_NAME` — название организации
  - `FISCAL_ADDRESS` — адрес точки
  - `FISCAL_API_URL` — URL API провайдера
  - `FISCAL_API_KEY` — API ключ
  - `FISCAL_CALLBACK_URL` — URL для callback
- Формирование чеков в формате 54-ФЗ

**Критерий приёмки**: Чеки отправляются в облачную кассу и регистрируются в ФНС.

---

### 19.2. Автоматический чек при оплате

**Описание**: При успешной оплате заказа автоматически формируется и отправляется чек.

**Детали**:
- Срабатывает при получении webhook от банка со статусом `paid` или `captured`
- Чек содержит:
  - Наименование товаров (название блюда)
  - Количество каждого товара
  - Цена за единицу
  - Сумма по позиции (цена × количество)
  - НДС (если применимо)
  - Итого к оплате
  - ФИО/название продавца
  - ИНН продавца
  - Адрес точки
  - Дата и время чека
  - Признак расчёта (онлайн)
- Сохраняется `receipt_id` и `receipt_url` в заказе

**Критерий приёмки**: Чек отправляется автоматически при каждой успешной оплате.

---

### 19.3. Чек на возврат

**Описание**: При частичном или полном возврате формируется чек возврата.

**Детали**:
- Создаётся при выполнении refund через админ-панель
- Содержит ссылку на исходный чек (`receipt_id`)
- Указывается сумма возврата
- При частичном возврате — только возвращаемые позиции

**Критерий приёмки**: Чек возврата формируется корректно.

---

### 19.4. Отправка чека клиенту на email

**Описание**: После формирования чека отправлять его клиенту на email.

**Детали**:
- Если клиент указал email при заказе — отправка автоматически
- Используется ссылка из `receipt_url` или вложенный PDF
- Email отправляется через NotificationService

**Критерий приёмки**: Клиент получает чек на email.

---

### 19.5. Статус чека в админ-панели

**Описание**: Отображение статуса фискализации в деталях заказа.

**Детали**:
- Показать статус: `pending` (не отправлен), `sent` (отправлен), `completed` (подтверждён), `error` (ошибка)
- Показать ID чека (`receipt_id`)
- Показать ссылку на чек (`receipt_url`)
- Показать ошибку фискализации (если есть)
- Кнопка "Повторить отправку" при ошибке

**Критерий приёмки**: Статус чека виден в админ-панели.

---

### 19.6. Ручная отправка чека

**Описание**: Возможность вручную отправить чек для заказа.

**Детали**:
- Кнопка "Отправить чек" в деталях заказа
- Доступно для заказов с успешной оплатой
- После отправки обновить статус

**Критерий приёмка**: Чек отправляется вручную.

---

### 19.7. Обработка ошибок фискализации

**Описание**: Корректная обработка и логирование ошибок.

**Детали**:
- При ошибке API — записать в `fiscal_error`
- Повторные попытки (3 попытки с экспоненциальной задержкой)
- Уведомление администратору при критических ошибках
- Сохранение `fiscal_status` = 'error'

**Критерий приёмки**: Ошибки фиксируются и понятны администратору.

---

### 19.8. Проверка статуса чека

**Описание**: Периодическая проверка статуса отправленного чека.

**Детали**:
- При получении callback от ПКФ — обновление статуса
- Ручная проверка через админ-панель
- Максимум 3 проверки с интервалом

**Критерий приёмки**: Статус чека обновляется корректно.

---
